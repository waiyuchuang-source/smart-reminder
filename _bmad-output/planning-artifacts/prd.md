# Product Requirements Document - smart-reminder

**Author:** Shixiansheng
**Date:** 2026-04-30

## Executive Summary

智能提醒助手旨在彻底改变工业时代机械式的“任务催促”模式，打造一个具有高度同理心与目标感的 PC Web 端“学习搭子”。针对当前由于缺乏有效触达导致 40% 活跃学生未完成当日任务、整体完成率仅 60% 的业务痛点，本项目通过行为干预机制提供场景化微引导 (Micro-nudges)，核心业务目标是将每日任务完成率稳步提升至 75%，并将提醒的触达打开率从 8% 提升至 15%。

### What Makes This Special

- **基于同理心与目标感的设计**：在 PC 端界面中，以非打扰式的常驻小组件 (Widget) 展现学习进度。系统根据学生的状态标签（如连胜、落后、被动型），动态匹配大模型生成的情绪化关怀文案，将冷冰冰的“任务待办”转化为带有沉浸感与成就感的互动体验。
- **极致的成本控制架构**：摒弃昂贵的实时大模型并发调用和低到达率的桌面端原生推送（Desktop Push）。采用定时特征提取、大模型异步批量预生成文案的轻量级架构；针对 25% 连续未登录的流失用户，降维采用微信模板消息或短信进行低成本的站外召回。

## Success Criteria

### User Success
- **情绪破冰与同理心**：被动型学生不再对提醒产生抗拒感（零投诉/极低关闭通知意愿），在看到“学习搭子” Widget 后主动点击“开始任务”或“查看进度”的转化率显著提升。
- **目标达成感**：用户能够通过视觉反馈清晰感知到自己的连胜记录和当日进度，形成“为目标而战”的积极心态。

### Business Success
- **核心转化指标**：每日任务完成率从当前的 60% 提升至 75%。
- **触达激活指标**：提醒小组件/推送的交互打开率从 8% 提升至 15%。
- **防流失指标**：“连续 3 天以上未打开”的濒临流失用户比例（当前 30%）有效下降。

### Technical Success
- **极低成本架构**：大模型异步批量预生成文案的设计，确保单次用户曝光的 AI API 边际成本几乎为零，不因 DAU 扩大而导致账单失控。
- **高可用与到达率**：PC Web 站内 Widget 的渲染曝光率达到近乎 100%（成功避开了系统级 Push 仅 60% 的到达率瓶颈）。

### Measurable Outcomes
- 上线 1 个月内，完成率数据显著上涨；上线 3 个月内，达成 75% 的最终业务指标。
- Widget 每日曝光量、各文案模板点击率的漏斗数据完整且可监控。

## Product Scope

### MVP - Minimum Viable Product
- **目标感 Widget (前端)**：PC Web 端常驻的“学习搭子”界面，支持展示基础进度条、连胜天数和当日 AI 关怀文案。
- **异步文案匹配引擎 (后端)**：基于定时任务 (Cron Job) 提取用户状态标签（如被动型、活跃时段），并从 AI 预先生成的静态模板库中抽取匹配文案，无实时 API 阻力。

### Growth Features (Post-MVP)
- **低成本站外召回**：针对那 25% 完全不登录网页的流失用户，接入微信模板消息或短信渠道，发送 AI 个性化召回文案。
- **动态 A/B 测试**：对不同语气（激励型、安抚型、幽默型）的文案进行自动化 A/B 测试与数据回流。

### Vision (Future)
- **实时全知 AI 导师**：不仅限于任务提醒，还能在学生做题卡壳时，直接在 Widget 内进行多轮对话、题目辅导和情绪疏导。

## User Journeys

### Journey 1: 林晓（被动型学生）- 温暖的破冰与连胜守护 (Primary User - Success Path)
- **背景 (Situation)**: 林晓是个容易分心的初二学生。晚上 7 点她打开了 PC 学习平台，但习惯性地先切去看了眼 B 站，把学习任务抛在了脑后。
- **冲突 (Pain)**: 过去的系统会在 8 点弹出一个刺眼的红色警告：“您还有 3 个未完成任务！”，这经常让她感到焦虑甚至反感，索性直接关掉网页。
- **旅程 (Journey)**:
  1. **开场 (Opening)**: 林晓切回平台页面，发现右下角安静地躺着一个“学习搭子” Widget。
  2. **发展 (Rising Action)**: Widget 没有标红警告，而是展示了一个即将闭合的环形进度条，旁边是 AI 生成的一句懂她的文案：“晓晓，今天就只差最后 10 分钟的阅读小测啦，做完就能安心看剧休息咯~”
  3. **高潮 (Climax)**: 这种低门槛、无压力的表述打动了她。她没有点击关闭，而是顺手点击了 Widget 上的“一键开始”按钮，直接进入了小测页面。
  4. **结局 (Resolution)**: 测验完成，Widget 弹出一个小烟花动画：“太棒了！你的连续打卡天数来到了 5 天！”林晓心满意足地合上电脑，不仅没有焦虑，反而获得了掌控感。

### Journey 2: 张博（自驱型学生）- 正向强化与惊喜感 (Primary User - Edge Case)
- **背景 (Situation)**: 张博是班里的学霸，属于那 40% 的自驱型用户。他通常一上线就会一口气把任务做完。
- **冲突 (Pain)**: 原有的提醒系统对他来说毫无用处，甚至显得很多余。
- **旅程 (Journey)**:
  1. **开场 (Opening)**: 早上 9 点，张博一上线就完成了所有任务。
  2. **发展 (Rising Action)**: 异步引擎捕获到了他的“极速通关”标签，自动为他分配了“早鸟/学霸”模板的赞美文案。
  3. **结局 (Resolution)**: Widget 变成了金色荣誉形态：“哇！你是今天全站前 5% 完成所有任务的效率大师！”张博微笑着截图发到了朋友圈。系统将原本无用的提醒转化为了强大的正向情绪价值。

### Journey 3: 王运营（系统运营/产品人员）- 掌控全局与调优 (Admin/Operations User)
- **背景 (Situation)**: 王运营需要对产品的“15% 任务完成率增长”负责。
- **冲突 (Pain)**: 她不知道哪些 AI 生成的文案语气（鼓励式、激将法、同理心）对转化率最有效，而且担心 AI 跑错方向。
- **旅程 (Journey)**:
  1. **开场 (Opening)**: 王运营登录后台的“智能提醒引擎看板”。
  2. **发展 (Rising Action)**: 她查看到昨晚的批量推送数据。数据显示，“同理心”标签下的文案点击率高达 18%，而“激将法”只有 5%。
  3. **高潮 (Climax)**: 她在后台调整了策略权重，让今晚的 Cron Job 异步任务在生成文案时，向“同理心”倾斜分配 80% 的流量。
  4. **结局 (Resolution)**: 她清晰地掌控着大模型的输出边界和转化效果，对完成本季度的 KPI 充满信心。

### Journey Requirements Summary

从以上旅程中，我们可以提炼出以下具体的能力需求（Capabilities）：
1. **前端交互层 (Widget)**：需要非打扰式的 UI 设计、平滑的展开动画、可视化的进度条，以及能直达任务的 Deep Link 按钮。
2. **后端状态机 (State Machine)**：需要准确捕捉用户的实时状态（刚上线、任务完成进度）和历史标签（连胜、早鸟）。
3. **文案匹配引擎 (AI Engine)**：需要能根据标签从预生成的 AI 模板库中下发对应的文案，而不是一刀切的硬编码文案。
4. **运营看板 (Admin Dashboard)**：需要记录不同文案模板的曝光、点击数据，并支持运营人员调整流量分配权重。

## Domain-Specific Requirements

### Compliance & Regulatory (合规与监管)
- **数据隐私保护**：严格遵守《未成年人网络保护条例》及相关教育 App 备案标准。向第三方大模型（如 Claude/OpenAI 等）请求生成文案时，必须对学号、真实姓名、学校等 PII (个人敏感信息) 进行脱敏，仅传输匿名化的行为标签。
- **内容安全与价值观审查**：面向未成年人的 AI 生成内容必须绝对安全。异步引擎生成的文案必须经过敏感词过滤（或自带的安全围栏），禁止出现消极、误导或不适宜未成年人的内容。

### Technical Constraints (技术约束)
- **大模型“幻觉”隔离**：为了防止 AI 虚构作业内容或弄错截止时间，大模型仅负责生成“情感化外壳”和“骨架模板”（例如：“太棒了！今天只差 {task_name} 啦！”）。具体的硬性业务数据（如任务名称、进度百分比）必须由前端或本地后端通过变量插值动态填入。
- **异步解耦架构**：教育系统在晚上 19:00-21:00 会有明显的晚高峰，AI 生成引擎必须是纯异步的 Cron 任务，绝不能因为外部 API 延迟而阻塞主业务流程的登录或页面渲染。

### Integration Requirements (集成需求)
- **状态同步系统**：需要与现有的 LMS（学习管理系统）紧密集成，能够只读获取用户的“今日任务列表”、“历史连胜天数”和“在线状态”这三个关键接口。

### Risk Mitigations (风险缓解)
- **文案降级兜底方案**：如果异步生成引擎崩溃，或者生成的文案未能通过安全审查，前端 Widget 必须能自动降级（Fallback）为预设的、安全的人工手写兜底文案库，确保功能可用性。
