## 一、背景和目标

### 1. 需求背景

我们是一款 AI 教育产品，学生需要在 App 中完成每日学习任务（如看课、做题等）。当前发现一个核心问题：**部分学生不会主动打开 App 完成当天任务，导致学习中断、学习效果下降。**

现有推送体系存在三个不足：

- **时机单一**：仅每天 18:00 统一推送"你还有学习任务未完成"，无法覆盖学生活跃的两个黄金时段（放学后 16:00-18:00 占 35%、晚饭后 19:00-21:00 占 45%）。
- **文案千篇一律**：对自驱型（40%）、被动型（35%）、流失型（25%）三类学生使用相同文案，缺乏个性化触达，推送打开率仅 8%。
- **缺乏疲劳度控制**：无频次管控和防打扰机制，容易对已完成任务的学生或高频触达用户造成骚扰，导致用户主动关闭通知权限。

### 2. 项目收益

- **任务完成率提升**：将每日任务完成率从当前的 60% 提升至 75%（+15pp）。
- **推送打开率提升**：推送打开率从 8% 提升至 15%+（+7pp）。
- **流失用户召回**：通过流失预警 + 家长学情周报，降低连续 3 天以上未登录用户的比例（从 30% 降至 20%）。
- **用户体验改善**：基于疲劳度控制减少无效推送，NPS 提升。

### 3. 覆盖用户

- 全量付费学生用户（约 30 万 DAU）
- 管理后台面向运营/产品人员

### 4. 竞品调研

| 维度       | 竞品 A（作业帮） | 竞品 B（猿辅导） | 本方案               |
| ---------- | ---------------- | ---------------- | -------------------- |
| 推送时机   | 固定时段推送     | 固定时段推送     | 个性化时机匹配       |
| 文案策略   | 统一模板         | 分学科模板       | AI + 多语气分群匹配  |
| 社交激励   | 无               | 排行榜           | 同桌机制 + 实时互动  |
| 疲劳控制   | 每日上限         | 无               | 多因子疲劳度模型     |
| A/B 实验   | 无               | 有               | 内置实验平台         |

### 5. 方案简述

- **智能时机匹配**：基于用户分群（自驱型 / 被动型 / 流失型）与行为画像，在最佳时机触发提醒，而非固定 18:00 统一推送。
- **AI 多语气文案引擎**：支持三种语气（同理心 / 激励 / 幽默），AI 生成 + 规则模板双模降级，按用户分群与历史 CTR 动态匹配最优文案。
- **双层频控 + 疲劳度模型**：全局冷却（15 分钟）+ 同类型冷却（1 小时）+ 每日上限（5 次），并引入三因子疲劳度评分（dismiss 率 × 0.5 + 近期密度 × 0.3 + 时段因子 × 0.2）做自适应降频。
- **社交激励体系**：同桌机制、队友动态、学情排行，利用社交压力驱动被动型和流失型用户回归。
- **流失预警与家长触达**：自动检测连续 4 天未登录用户，触发流失预警并支持发送家长学情周报。
- **运营后台与 A/B 实验**：可视化运营面板，支持权重调节、文案管理、频控配置、疲劳度监控及在线 A/B 实验。

## 二、名词说明

| 名词           | 说明                                                                     |
| -------------- | ------------------------------------------------------------------------ |
| Nudge          | 一次提醒触达动作，包含文案选择、时机判断、渠道投放的完整流程               |
| Segment（分群）| 用户分群标签：`active`（自驱型）、`passive`（被动型）、`at-risk`（流失型）|
| Tone（语气）   | 文案语气类型：`empathetic`（同理心）、`motivational`（激励）、`humorous`（幽默） |
| Fatigue Score  | 疲劳度评分（0-100），综合 dismiss 率、近期推送密度、时段因子计算           |
| Nudge Governor | 频率控制器，实现全局冷却期（15 分钟）、同类型冷却期（1 小时）、每日上限（5 次）|
| Copy           | 提醒文案模板，支持占位符 `{user_name}` `{task_name}` `{streak}` `{buddy_name}` 等 |
| A/B Experiment | 在线实验平台，支持对提醒策略的对照实验，基于 hash 分桶保证确定性分流     |
| Desk Mate      | 同桌机制中配对的学习伙伴，用于社交激励                                     |
| Churn Alert    | 流失预警，对连续 ≥ 4 天未登录的用户触发告警                              |

## 五、详细产品方案

### 1. 用户分群与时机匹配

| 模块     | 需求说明                                                                                                   |
| -------- | ---------------------------------------------------------------------------------------------------------- |
| 用户分群 | **分群规则**                                                                                                |
|          | • **自驱型（active）**：每天主动完成任务，约占 40%。                                                       |
|          | • **被动型（passive）**：需提醒才完成，约占 35%。支持用户设置"偏好提醒时间"。                              |
|          | • **流失型（at-risk）**：连续多日未活跃，提醒也不来，约占 25%。                                             |
|          | 分群存储于用户 `segment` 字段，支持运营手动调整或后续算法自动流转。                                         |
| 时机策略 | **自驱型**                                                                                                  |
|          | • 最小打扰：仅在 **21:00** 如仍有未完成任务时提醒一次。                                                    |
|          | • 其他时段返回："自驱型学霸无需打扰，如果 21:00 还没完成「{task_name}」，我再来提醒你~"                    |
|          | **被动型**                                                                                                  |
|          | • 优先使用用户设置的 `preferredReminderTime`，在该时间前 15 分钟内触发。                                    |
|          | • 若未设置或不在偏好时段：在两个活跃高峰窗口（16:00-18:00、19:00-21:00）内允许触发。                       |
|          | **流失型**                                                                                                  |
|          | • 全时段可触发（`isAppropriateTime = true`），最大力度尝试召回。                                            |
|          | • 配合社交激励和家长周报多通道触达。                                                                        |

### 2. AI 多语气文案引擎

| 模块         | 需求说明                                                                                                  |
| ------------ | --------------------------------------------------------------------------------------------------------- |
| 双模生成     | **模式 1 — LLM 模式**                                                                                     |
|              | • 调用大语言模型（配置 `OPENAI_API_KEY` 或兼容接口），按语气描述生成 30 字以内学习提醒文案。              |
|              | • 语气描述映射：                                                                                          |
|              |   - empathetic → "温柔共情，像知心朋友"                                                                  |
|              |   - motivational → "充满能量，像教练加油"                                                                |
|              |   - humorous → "轻松幽默，用有趣的比喻"                                                                 |
|              | • temperature = 0.95，maxTokens = 100，保证多样性。                                                       |
|              | **模式 2 — 规则降级**                                                                                     |
|              | • 当 LLM 不可用或调用失败时，从预定义模板池随机选取。每种语气预置 3 条模板。                                |
|              | • 模板支持占位符：`{user_name}` `{task_name}` `{task_duration}` `{streak}` `{buddy_name}`                 |
| 批量生成 API | • 接口：`POST /api/nudge/generate`                                                                        |
|              | • 参数：`tone`（语气类型）、`count`（生成数量）                                                          |
|              | • 生成后存入 NudgeCopy 表，供匹配引擎使用。                                                               |
| 文案管理     | • 每条文案包含：`id`、`tone`、`targetSegment`、`template`、`baseScore`（0-10 基础质量分）、`tags`（标签）  |
|              | • 标签体系：`efficient`（高效）、`streak`（连胜）、`social`（社交）、`competitive`（竞争）、`reward`（奖励）等 |

### 3. 文案智能匹配

| 模块         | 需求说明                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------- |
| 候选筛选     | 从 NudgeCopy 库中筛选候选文案：                                                                        |
|              | • `targetSegment` 匹配用户分群，或为 `all`                                                             |
|              | • 语气匹配分群偏好：active → motivational，passive → empathetic，at-risk → humorous                   |
|              | • 中等疲劳时额外过滤：仅保留 `baseScore ≥ 7` 的高质量文案                                              |
| 多因子打分   | 对候选文案逐一计算综合分数：                                                                            |
|              | • 基础分：`baseScore`（0-10）                                                                         |
|              | • 分群匹配：+3 分                                                                                     |
|              | • 语气偏好匹配：+2 分                                                                                 |
|              | • 任务时长 > 30min 且带 `efficient` 标签：+1 分                                                        |
|              | • 连胜 > 7 天且带 `streak` 标签：+1.5 分                                                               |
|              | • 有同桌且带 `social` 标签：+1 分                                                                      |
|              | • 同桌在线且带 `social` 标签：+2 分                                                                    |
|              | • 同桌完成率 > 50% 且带 `competitive` 标签：+1.5 分                                                    |
|              | • 随机抖动：+random(0,1)，防止总是选同一条                                                             |
| 占位符渲染   | 选中文案后，将模板中的占位符替换为真实值：                                                                |
|              | `{user_name}` → 用户名，`{task_name}` → 首个未完成任务名，`{streak}` → 连胜天数，`{buddy_name}` → 同桌名 |

### 4. 双层频控机制

| 模块             | 需求说明                                                                                           |
| ---------------- | -------------------------------------------------------------------------------------------------- |
| 全局冷却         | 同一用户两次提醒间隔 ≥ **15 分钟**（可配置 `globalCoolingMinutes`）                                 |
| 同类型冷却       | 同一类型的提醒间隔 ≥ **1 小时**（可配置 `typeCoolingMinutes`）                                     |
| 每日上限         | 同一用户每日最多收到 **5 次**提醒（可配置 `maxDailyNudges`）                                        |
| 双端实现         | • **服务端**：API Route 内使用内存 Map 做频控判断（`/api/nudge/frequency`）                         |
|                  | • **客户端**：localStorage 降级存储 `nudge_history`，保留最近 50 条事件                             |
|                  | • 优先使用服务端判断，服务端不可达时降级到客户端                                                     |
| 配置中心         | 存储于 `SystemConfig` 表（key = `frequency_config`），运营后台可动态调整，带 60 秒内存缓存           |

### 5. 疲劳度模型

| 模块           | 需求说明                                                                                             |
| -------------- | ---------------------------------------------------------------------------------------------------- |
| 三因子模型     | **因子 1 — dismiss 率**（权重 0.5）                                                                   |
|                | • 计算方式：近 7 天内 `dismissed / (viewed + dismissed)`                                              |
|                | • 反映用户对提醒的厌烦程度                                                                            |
|                | **因子 2 — 近期推送密度**（权重 0.3）                                                                  |
|                | • 计算方式：过去 2 小时内收到的提醒总数，上限 Cap = 6                                                  |
|                | • 反映短期内是否被过度触达                                                                            |
|                | **因子 3 — 时段因子**（权重 0.2）                                                                     |
|                | • 23:00-06:00 → 1.0（深夜，不应打扰）                                                                |
|                | • 06:00-08:00 / 22:00-23:00 → 0.5（早晚边界）                                                       |
|                | • 其他时段 → 0.0                                                                                     |
| 分级策略       | 总分 = Σ(因子 × 权重 × 100)，范围 0-100                                                               |
|                | • **低疲劳（< 40）**：正常推送                                                                        |
|                | • **中疲劳（40-69）**：仅推送高质量文案（baseScore ≥ 7）                                              |
|                | • **高疲劳（≥ 70）**：完全暂停推送，返回"检测到你可能有些疲惫，先休息一下吧"                           |
| 阈值可配置     | `fatigueThresholdHigh`（默认 70）和 `fatigueThresholdMedium`（默认 40）可在运营后台动态调整            |

### 8. 流失预警

| 模块         | 需求说明                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------- |
| 检测规则     | • 扫描所有用户，计算 `(当前时间 - lastActiveAt)` 的天数。                                             |
|              | • 阈值：连续 ≥ **4 天**未登录即标记为流失风险（`CHURN_THRESHOLD_DAYS = 4`）。                         |

|              | • 支持一键"通知家长"操作，标记已通知状态和通知时间。                                                  |

| 模块         | 需求说明                                                                                             |

## 八、A/B 实验需求

### 实验 1：语气策略对比

| 维度     | 说明                                                                     |
| -------- | ------------------------------------------------------------------------ |
| 实验名称 | tone_strategy_v1                                                         |
| 实验类型 | tone                                                                     |
| 变体     | A：同理心为主（empathetic 60%, motivational 30%, humorous 10%）          |
|          | B：激励为主（motivational 60%, empathetic 30%, humorous 10%）            |
|          | C：均匀分布（各 33.3%）                                                  |
| 核心指标 | 提醒点击率（CTR）、任务完成率、7 日留存率                                |
| 分桶方式 | userId + experimentId 确定性 hash                                        |
| 样本量   | 每组 ≥ 10,000 用户                                                       |
| 实验周期 | 14 天                                                                    |

### 实验 2：推送时机对比

| 维度     | 说明                                                                     |
| -------- | ------------------------------------------------------------------------ |
| 实验名称 | timing_strategy_v1                                                       |
| 实验类型 | timing                                                                   |
| 变体     | A：固定 18:00 推送（对照组 / 现有策略）                                  |
|          | B：分群时机策略（实验组 / 本方案）                                        |
| 核心指标 | 推送打开率、任务完成率、dismiss 率                                        |
| 分桶方式 | 同上                                                                     |
| 样本量   | 每组 ≥ 15,000 用户                                                       |
| 实验周期 | 14 天                                                                    |

### 显著性检验

- 使用 Z 检验（双尾），显著性水平 α = 0.05
- 检验公式：`Z = (p1 - p2) / sqrt(p*(1-p)*(1/n1 + 1/n2))`
- 后台自动计算 p-value 和置信区间，辅助运营决策
